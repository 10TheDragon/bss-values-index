Item Value
getAvg(item) — the foundation of every other calculation
getAvg used everywhere

Every value displayed and compared in the tool comes from this function. It prefers the site's own smoothed average, falling back to a midpoint if unavailable.

// Priority 1 — use bssmvalues.com's smoothed graph value if cached_graph_value > 0: return cached_graph_value // Priority 2 — midpoint of min/max range return (value_min + value_max) / 2

If both value_min and value_max are missing or non-numeric, the item is treated as unpriced and excluded from value-based calculations.

Tier System
getTier(avg) — community-defined value brackets

Tiers are community labels, not official game categories. The tool walks this table top-to-bottom and assigns the first tier whose range contains the item's avg value.

Trading Calculator
Verdict labels, trade type detection, and adds suggestions
Verdict Thresholds
diff = yourTotal − theirTotal pct = |diff| / theirTotal × 100 pct ≤ 5% → Fair Trade ✓ pct ≤ 15% AND diff < 0 → Close — LB (suggests adds) diff > 0 → OP (you overpaid) diff < 0 → LB (you underpaid)

Both the raw value difference and the percentage are shown simultaneously. For example: OP +3.50 (+18.2%).

Trade Type Detection UP / DG
yourItems > 1 AND theirItems = 1 → ⬆ UP (Upgrading) yourItems = 1 AND theirItems > 1 → ⬇ DG (Downgrading) both sides > 1 → ↔ Multi-item trade
💡 In an UP trade expect to OP slightly. In a DG trade require OP from them. This is a community convention, not enforced by the tool.
Adds Suggestion

When you're LB by ≤15%, the tool finds items from the full catalogue that could close the gap:

gap = |diff| candidates = items where avg ≤ gap × 1.2 sorted by: |item.avg − gap| ascending show top: 4 results

Trade Success %
calcTradeSuccess(combo, targetValue) — core probability estimate

Estimates the likelihood a trade offer gets accepted in the BSS trading hub. Output is always between 5 and 95 — a trade can never be guaranteed or impossible.

n = number of items you're offering total = sum of avg values of your items base = 60 // ~60% baseline acceptance rate in the hub // Factor 1: Value closeness (0–60 pts) valueDiff = |total − targetValue| / targetValue valueScore = max(0, 1 − valueDiff × 2.5) × 60 // Factor 2: Overpay bonus (0–+15 pts) opRatio = (total − targetValue) / targetValue opBonus = if OPing: min(15, opRatio × 40) // Factor 3: Item count penalty (–5 per extra item) countPenalty = (n − 1) × 5 // Factor 4: Tier distance penalty (–8 per tier jumped) avgComboTier = average tier index of your items targetTierIdx = tier index of target item (0=Legendary … 7=Basic) tierDistance = max(0, targetTierIdx − avgComboTier) tierPenalty = tierDistance × 8 // Final score raw = base + valueScore + opBonus − countPenalty − tierPenalty result = clamp(round(raw), 5, 95)
FactorMax contributionDirection
Base rate60 ptsAlways added
Value closeness+60 ptsReduces as values diverge
Overpay bonus+15 ptsScales with how much you OP
Item count−5 per extra item1 item = 0 penalty, 4 items = −15
Tier distance−8 per tier jumpedSame tier = 0, 3 tiers = −24
⚠️ This is an estimate based on BSS trading conventions, not a real probability derived from actual trade data. Use it as a guide, not a guarantee.

Demand Adjustment
calcTradeSuccessAdj — applied on top of base success when Sort Mode is not Value

Takes the base success % and adjusts it based on demand levels of what you're giving vs what you're asking for.

Sort ModeFormula
💰 Value No adjustment — base score returned unchanged
⚖️ Both base
− (targetDemand / 5) × 15  (up to −15 for high-demand target)
+ (avgYourDemand / 5) × 12  (up to +12 for high-demand offer)
📊 Demand 50 + (avgYourDemand − targetDemand) × 12
(value math thrown out — only demand gap matters)
💡 Why this matters: Offering high-demand items means the other person wants what you're giving — they're more likely to accept. Asking for high-demand items means you're competing with many other people LFing the same thing — harder to acquire.

Upgrader
Target pool filtering, combo generation, and sorting
Target Pool Filter

Items from the full catalogue are shown as upgrade targets only if their value is within ±10% of your combo's total value:

comboTotal × 0.90 ≤ target.avg ≤ comboTotal × 1.10
Combo GenerationallSubsets()

Every possible non-empty combination of your items is generated using bit masking:

// For each number from 1 to 2ⁿ − 1: // treat it as a bitmask where each bit = one item for mask = 1 to (2^n − 1): include item[i] if bit[i] of mask is set max items: 8 (2⁸ − 1 = 255 combos)

Capped at 8 items to prevent browser lag. Combos that produce the same target are deduplicated, keeping only the highest success % per target.

Wishlist
Coverage bar, gap value, and trade path matching
Coverage Bar
targetTotal = sum of avg values of all wanted items pct = min(1, myTotal / targetTotal) // Colour thresholds pct ≥ 0.90 → green (can afford) pct ≥ 0.60 → amber (getting close) pct < 0.60 → red (significantly short)
Gap Value

This determines what value the tool searches for in singles and pairs — either the full target or just the remaining gap:

if you have items AND you're short of target: gapVal = targetTotal − myTotal // find what you still need else: gapVal = targetTotal // find something for the full target

Trasher Score
calcTrashScore(item, invAvg) — 0 to 100, higher = more trashable

invAvg = average value of all items in your drop zone that have a numeric value (unpriced items are excluded from this average).

Sort ModeFormulaWhat it measures
💰 Value round(min(100, max(0, (1 − avg/invAvg) × 70 + 30))) How far below your inventory average the item is
📊 Demand round((1 − demand/5) × 100) How rarely this item is traded or requested in the hub
⚖️ Both round((1 − demand/5) × 60 + min(40, (1 − avg/invAvg) × 40)) 60% demand + 40% value blend
// Score interpretation score ≥ 61 → TRASH (red) score ≥ 31 → MAYBE (amber) score < 31 → KEEP (green)
⚠️ Limited / seasonal items are never recommended for trash — the Recommender protects these regardless of their score.

Portfolio Health
calcPortfolioHealth() — 0 to 100 overall score, shown in Recommender

Four factors, each scored 0–100, then averaged into a single overall score.

FactorFormulaWhat it measures
Avg Demand (weightedAvgDemand / 5) × 100 How tradeable your inventory is on average
Diversity min(1, uniqueTierCount / 5) × 100 How many different value tiers are represented
Liquidity max(0, 1 − spread) × 100
spread = weighted mean absolute deviation / total value		 How close your items are to each other in value (tightly clustered = liquid)
Value Density min(1, avgValuePerItem / 10) × 100 Average value per item (caps at avg 10)
overall = round((demandScore + divScore + liqScore + densityScore) / 4) overall ≥ 70 → Healthy overall ≥ 45 → Fair overall < 45 → Needs Work

Market Overview
Price change detection between refreshes

Before every data refresh, current values are snapshotted. After the refresh, each item is compared to its snapshot to detect significant changes.

// An item appears as a gainer or loser only if: |current − previous| / previous ≥ 0.5% // The same threshold triggers the flash animation on item cards

Items below this threshold are considered noise and not shown as movers. In Value mode only demand ≥3 items appear in the Most Demanded strip; in Demand mode all items are shown.

Recommender
Rule-based per-item recommendations across three modes

No numeric formula — applies conditional rules to each item in your inventory. Limited items (marked _static, or avg ≥50 + demand ≥3) are protected from trash/trade-away recommendations in all modes.
